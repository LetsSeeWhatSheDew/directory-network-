# Hero Deal Card Rendering Audit — 2026-04-21

**Context:** Commit f5d4536 reworked the homepage hero card to include four new data fields — regular/sale price comparison, open-now indicator, distance-from-you, and DealFreshnessBadge. This audit verifies each field is wired end-to-end on production.

## Architectural note (important for verification)
`HeroDealCard` is marked `"use client"` ([app/components/HeroDealCard.tsx:1](app/components/HeroDealCard.tsx:1)). On SSR it renders a skeleton; populated data only appears after hydration via the useEffect that calls `/api/deals/recommend`. **Therefore curl against `/` will show the skeleton, not the enriched card.** Field presence must be verified at three layers:
1. Component source — does it render the field?
2. API response — does it return the data the field needs?
3. SSR HTML — does the skeleton + CSS ship?

### Latent bug — `verified_at` not projected by view

The `/api/deals/recommend` route does `select=*` against the `active_deals_with_listings` view, but the view ([sql/migrations/2026-04-20-fix-active-deals-view.sql:49](sql/migrations/2026-04-20-fix-active-deals-view.sql:49)) does **not** include `d.verified_at` in its projection. Today this has no visible impact — all 56 active deals have `verified_at = NULL`, so the badge correctly renders "Verification pending" anyway. But when verification data starts landing (scraper, manual audit, etc.), the hero card will keep rendering "Verification pending" forever, because the field never reaches the client.

**Fix (Code + Cowork coordination):**
- Cowork: add `d.verified_at` to the view projection in a follow-up migration.
- Code: no change required once the view is updated — the field flows through automatically (`select=*`).

**Severity:** P2 — no visible regression today; blocks future freshness UX.
